# Encrypted MCP Payload Detection (Without Decryption)

## Data Generation

We will be using docker compose to manage 4 containers:

1. mcp-client - This is the container having the MCP client which communicates with the MCP server
2. mcp-server - This is the container having the MCP server, connected to a sqlite server to run queries
3. noise-generator - This is the container having a shell script which calls random APIs to download data and get random data.

This architecture is made to ensure that the data generated resembles reali-life data.

### mcp-client

This has 3 files:

1. client.py:
    - The actual MCP client code which initiates the https connection with the MCP server.
    - Since the certificate for https is self signed by the server (we do not have the 3rd party signed certs), this will siable the warning and accepts the cert for initiating the connection
    - Connects to the proxy socat which in turn initiates a http connection with the server (because of the untrusted certificates, making https connection from client to server is infeasible as the sdk does not support unverified certificates) MCP server using SSE (Server Sent Events)
    - Chooses a random number of queries and after they have been executed, it will reset the connection, i.e. make a new one
    - It also has random "think time" which somewhat simulates the actual latency of an llm thinking before responding.
    - Since the number of queries is random, the flow metadata will also differ from flow to flow
2. run.sh:
    - This helps us in starting the socat proxy which initiates a connection to the MCP server on the behalf of the MCP client
3. Dockerfile:
    - USed to build the container

### mcp-server

This has 3 files:

1. server.js:
    - This has the tool needed by the LLM for the response
    - It has connected to a sqlite database initialised with data (read the mcp-server/Dockerfile)
    - It has the SSE endpoint to which the socat proxy will connect to on behalf of the client
    - This also has random latecies to simulate real world network traffic
2. Dockerfile:
    - This is used to build the container and to initialise the db with random data resembling a real life database
    - This also generates the certificate needed for the HTTPS connection
3. package.json:
    - To download node modules

### noise-generator

This one is still primitive as it only downloads 10-100MB of data through different HTTPS connections to different sites and gets small json data from similar sites to keep the variation. But since the size differ so much there tends to be bias.

### network-tap

This container acts as a passive packet sniffer, silently monitoring and recording the traffic generated by the other containers without interfering with their operations.

Key Components:

- *The Image (`nicolaka/netshoot`)* This is an industry-standard, lightweight Docker image specifically built for network troubleshooting. It comes pre-installed with powerful networking binaries like `tcpdump`, `tshark`, and `nmap`, saving us from having to build a custom sniffing image from scratch.

- *The Network Mode (`network_mode: host`)* This is the most critical setting. By default, Docker isolates containers on their own virtual bridge networks, meaning a standard container can only see its own traffic. By elevating it to `host` mode, the container gains access to the host machine's entire networking stack, allowing it to "see" the virtual bridge interfaces Docker created for your MCP server and client.

- *The Sniffing Command (`tcpdump -i any...`)*
  - `-i any`: Tells `tcpdump` to listen on every available network interface. This is necessary because Docker dynamically generates randomized names for its virtual bridges (e.g., `br-a1b2c3d4`), and we want to capture the traffic regardless of what the bridge is named.
  - `'tcp port 8443 or tcp port 443'`: This is a Berkeley Packet Filter (BPF). It instructs the sniffer to drop all the useless background Docker chatter (like DNS queries or ARP broadcasts) and specifically grab your encrypted MCP traffic (`8443`) and your external internet noise (`443`).
  - `-w /capture/dataset.pcap`: Writes the raw binary packet data directly to a file instead of flooding the terminal console.

- *The Volume Mount (`./capture:/capture`)* Docker containers are ephemeral; if the container stops, the data inside it vanishes. This volume maps a folder on your physical hard drive directly to the container's capture directory, ensuring that your massive `.pcap` file safely persists on your host machine for Phase 2 processing.
